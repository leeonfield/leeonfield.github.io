时间：2017-7-18
坐标：北京
天气： 中雨转晴

我们组的 case 越来越多，老大把一个小项目给我负责，吩咐不懂的就去问。这是我第一次独立完成一个项目从始至终的开发，很多重要的细节都是第一次经历，所以过程一直在踩坑。昨晚和超艺联调到十一点四十，提测，打车回家，所以现在看到的这篇 llog 是我明天中午写的，有点累。

这里提几个认为需要关注的坑，以作纪念：

  1. 进行 需求评审时，对于模棱两可的点进行确认，并且付诸文字，留下凭据，对于实现难度比较大的功能，沟通，要么减需求，要么加时间。不要吝啬对需求的质疑。
  2. 对于模块用到的技术以及接口，需要提前调研清楚，不明白的地方及时询问或者联系 RD，这会影响后续开发预估时间。
  3. 评审确定后，向 UI 拿设计稿，对因为内容可能造成的排版问题（如字段长度过长造成的折行问题），及时提出质疑，并且联系 PM 进行沟通，以防后续 UI 变动。
  4. 功能需求优先级高于页面需求，功能需求涉及到与 RD 的联调过程，这部分需要实现确定。举个例子，项目的页面开发完了，但只剩一天的时间用于和 RD 联调，但是工作量是两天，这种情况会导致 RD 和你一起加班。但先将功能模块调通，后续页面开发自由度较高，可控性也较高。

与昨天写的 llog 「熟悉流程」类似，熟悉了流程，明白了过程的规则，才是游戏开始的关键。若不是老大的放手，我现在行事可能还是畏首畏尾。这两天过得很艰难，但是学到了东西，真正开始入门。付出是有价值的。

日子就这样一天天过去，今天是周二，但我已经预见了这一周即将过去，岁月如流水，一去不回头。
